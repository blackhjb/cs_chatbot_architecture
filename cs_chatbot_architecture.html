1. 시스템 개요

KareChat CS Agent는 병원 FAQ/CS 기반 AI 챗봇 시스템입니다.
사용자 질문을 분석하여 FAQ/CS 매칭, LLM 기반 응답 생성, 실시간 정보 조회를 수행합니다.

실시간 정보병원 연락처, 주소 등 실시간 정보 응답

2. 전체 시스템 아키텍처

정상 처리 흐름 (FAQ 매칭 성공)

사용자 질문이 FAQ에 고신뢰로 매칭되어 Gemini 호출 없이 즉시 응답하는 최적 경로입니다.

    라우터->>백오피스: 19. FAQ 목록 조회 (5분 캐시)

    라우터->>시맨틱: 21. 시맨틱 매칭 (Bi-encoder Top-10)
    Note over 시맨틱: KiwiPy 형태소 분석 → 임베딩 → 코사인 유사도
    시맨틱-->>라우터: 22. 상위 10개 후보 + 점수
    라우터->>크로스: 23. 재순위 요청 (Cross-encoder Top-3)
    Note over 크로스: Query-Document 쌍 직접 분류
    크로스-->>라우터: 24. 재순위된 Top-3 + 정밀 점수
    라우터->>라우터: 25. 통합 점수 산출 (키워드 40% + 시맨틱 60%)
    alt 고신뢰 매칭 (score >= 0.92)
        라우터-->>오케: 26. FAQ 직접 응답 (Gemini 생략)
        오케->>핸들러: 27. DirectFAQHandler 실행
        핸들러->>응답조립: 28. FAQ 원문 + 병원 안내 + 추천 질문
        응답조립->>세션: 29. 턴 히스토리 저장
        세션->>Redis: 30. 세션 업데이트
        응답조립-->>사용자: 31. 응답 반환 (gemini_skipped: true)
        Note over 사용자,응답조립: 응답 시간: 200~500ms
    end

의도 분류 경로 (FAQ 매칭 부족 → 후보 풀 준비 → Gemini에 전달하여 최종 판단)

FAQ 빠른 매칭에서 신뢰도가 부족할 때, 임베딩 데이터(FAQ 질문/답변/점수) + 사용자 질문을 Gemini에 넘겨서 FAQ ID를 최종 선택하는 경로입니다.

        시맨틱-->>의도: 7. 확장된 후보 풀 병합
        Note over 의도: 기존 후보 + 추가 후보 합쳐서 Gemini에 전달

    Note over 의도: Gemini는 이 enum 안에서만 FAQ ID 선택 가능
    의도->>Gemini: 10. 의도 분류 요청 (질문 + FAQ 후보 데이터 + enum 제약)
    Note over Gemini: flash-lite, temperature: 0.1, timeout: 15초
    alt Gemini 응답 성공

    else Gemini 타임아웃/오류

        FAQ서비스-->>폴백: 16. 추가 후보 반환

        혼합핸들러->>Gemini: 20c. 혼합 응답 생성 (flash-lite, Grounding)
        Gemini-->>혼합핸들러: 21c. 결합 응답 반환
        혼합핸들러->>응답조립: 22c. 응답 전달
    else Direct Answer (FAQ 없음)
        의도->>직접핸들러: 19d. Gemini 직접 응답
        직접핸들러->>Gemini: 20d. 답변 생성 (flash, Grounding, temp: 0.0)
        Gemini-->>직접핸들러: 21d. 생성 응답 반환
        직접핸들러->>응답조립: 22d. 응답 전달

FAQ 후보 풀 준비 → Gemini 전달 상세

전처리(Bi-encoder/Cross-encoder)에서 고신뢰 매칭을 못 찾았을 때, 임베딩 데이터 + 질문을 Gemini에 넘겨서 FAQ ID를 최종 선택합니다.

단계트리거 조건동작Gemini에 전달되는 데이터
1차: Rule-based 후보RuleBasedRouter에 top_candidates 존재상위 3~5개 후보 그대로 사용FAQ 질문+답변+combined_score+semantic_rate
2차: Semantic Fallback1차 후보가 비어있을 때SemanticMatcher로 Top-K 재검색Bi-encoder 유사도 상위 후보
3차: Low-grounding Widen후보 pool 전체 신뢰도가 낮을 때Top-K를 확장하여 더 넓은 범위 검색기존 후보 + 확장 후보 병합

Gemini 할루시네이션 방지 장치:

  • JSON Schema enum: 후보 FAQ ID만 enum으로 제한 → enum 밖 ID 출력 불가
  • FaqIdValidator: 가상 FAQ ID 감지 시 제거
  • Multiple FAQ Guard: LLM이 후보 밖 FAQ 추가 차단

3. 요청 처리 파이프라인

사용자 메시지가 수신되면 아래 단계를 순차적으로 거쳐 응답이 생성됩니다.

flowchart LR
    INPUT["사용자 메시지\nPOST /api/chat"] --> ADMIT["Admission Control\n(동시 요청 제한)"]
    ADMIT --> SAFETY["안전성 검사\n(PII/부적절)"]
    SAFETY --> TYPO["오타 보정\n(Gemini)"]
    TYPO --> ALIAS["도메인 별칭 확장\n(병원명 정규화)"]
    ALIAS --> HISTORY["세션 히스토리\n로드 (최대 7턴)"]
    HISTORY --> RULE["RuleBasedRouter"]
    RULE --> SEM["SemanticMatcher\n(Bi-encoder)"]
    SEM --> CROSS["CrossEncoder\n(Re-ranker)"]
    CROSS --> THRESHOLD["임계값 평가"]
    THRESHOLD -->|"고신뢰 FAQ 매칭\n(score >= 0.92)"| FAQ_H["FAQHandler\n(단일 FAQ)"]
    THRESHOLD -->|"매칭 부족"| INTENT_SVC["IntentService\n(Gemini LLM 호출)"]
    INTENT_SVC --> REMATCH["SemanticRematcher\n(보정)"]
    REMATCH --> FALLBACK["FallbackReranker\n(재순위)"]
    FALLBACK -->|FAQ| FAQ_H
    FALLBACK -->|Multiple FAQ| MULTI_H["MultipleFAQHandler\n(복수 FAQ)"]
    FALLBACK -->|Mixed| MIXED_H["MixedHandler\n(FAQ+실시간)"]
    FALLBACK -->|Direct Answer| DIRECT_H["DirectAnswerHandler\n(직접 응답)"]
    FALLBACK -->|Customer Service| CS_H["CustomerServiceHandler\n(상담사 연결)"]
    FALLBACK -->|Inappropriate| SAFETY_H["SafetyRouter\n(부적절 차단)"]
    FAQ_H --> BUILDER["ResponseBuilder\n(병원 안내 추가)"]
    MULTI_H --> BUILDER
    MIXED_H --> BUILDER
    DIRECT_H --> BUILDER
    CS_H --> BUILDER
    SAFETY_H --> BUILDER
    BUILDER --> TRANSFORMER["ResponseTransformer\n(LLM 다듬기)"]
    TRANSFORMER --> DISCLAIMER["DisclaimerManager\n(면책 문구)"]
    DISCLAIMER --> SUGGEST["RAGService\n(추천 질문)"]
    SUGGEST --> SAVE["SessionService\n(히스토리 저장)"]
    SAVE --> RETURN["API Response\n(JSON)"]

단계별 핵심 동작

단계핵심 동작평균 소요 시간
1. 전처리PII 검출, 오타 보정(Gemini), 병원명 정규화, 세션 로드200~500ms
2. FAQ 빠른 매칭Bi-encoder 유사도 → Cross-encoder 재순위 → 임계값 판단100~300ms
3. 의도 분류Gemini LLM 호출로 6가지 의도 분류 (FAQ 빠른 매칭 실패 시)1~3초
4. 핸들러 실행의도별 응답 생성 (필요시 Gemini 추가 호출)1~3초
5. 응답 조립병원 안내 추가, 면책 문구, 추천 질문 생성50~100ms
6. 응답 반환JSON 응답 전송즉시
FAQ 빠른 경로: 시맨틱 매칭 점수가 임계값(0.92) 이상이면 Gemini 호출 없이 즉시 FAQ 응답 → 3~5초 절약

4.2 SemanticMatcher (Bi-encoder 시맨틱 매칭)

파일: src/matching/semantic_matcher.py

항목상세
모델jhgan/ko-sroberta-multitask (한국어 특화 Sentence Transformer)
방식Bi-encoder: 질문/FAQ 각각 독립 임베딩 → 코사인 유사도 비교
Top-K상위 10개 후보 반환 (SEMANTIC_ANSWER_TOP_K)
캐싱인메모리 + Redis 분산 캐시 (멀티 워커 간 공유)
동기화매일 1회 FAQ 임베딩 자동 갱신 (Background Task)
모델 로딩모듈 레벨 싱글톤 (Gunicorn --preload로 COW 최적화)

동작 흐름:

  1. 사용자 질문을 KiwiPy 형태소 분석기로 정규화 (명사, 동사, 보조동사 추출)
  2. 정규화된 질문을 Sentence Transformer로 임베딩 벡터 생성
  3. 사전 계산된 FAQ 임베딩과 코사인 유사도 계산 (Top-10 추출)
  4. 질문 유사도 0.5~0.9 구간: 추가로 답변(Answer) 유사도도 계산하여 보정
  5. 짧은 질문 (< 20자)은 임계값을 낮춰 매칭 기회 확대

Calibration (점수 보정): 원시 코사인 유사도를 실제 신뢰도 스케일로 변환하여 임계값 판단의 정확도를 높임

4.7 IntentService (의도 분류)

파일: src/services/intent_service.py

Gemini LLM을 호출하여 사용자 질문의 의도를 분류합니다. FAQ 빠른 매칭이 실패했을 때만 호출됩니다.

분류 과정:

  1. Bi-encoder 상위 후보를 프롬프트에 포함하여 Gemini에 전달
  2. JSON 스키마 기반 구조화된 응답으로 의도 + FAQ ID + 신뢰도 반환
  3. 혼동 쌍(Confusion Pair) 감지: 상위 FAQ들이 의미적으로 유사하면 경고 → 더 정밀한 판단 유도
  4. 짧은 질문 특별 처리: 20자 미만 질문은 별도 임계값 적용
  5. 모호한 키워드 감지: 다의어 탐지 시 Gemini 강제 호출

4.8 Gemini 폴백 전략 (Fallback & Resilience)

전처리(Bi-encoder/Cross-encoder)에서 고신뢰 FAQ를 못 찾았을 때, 임베딩 데이터 + 질문을 Gemini에 넘겨서 FAQ를 찾아오는 경로와, Gemini마저 실패했을 때의 복구 전략입니다.

정상 경로 (Gemini에 FAQ 후보 전달):

단계트리거 조건동작Gemini에 전달되는 데이터
1차: Rule-based 후보RuleBasedRouter에 top_candidates 존재상위 3~5개 후보 그대로 사용FAQ 질문+답변+combined_score+semantic_rate
2차: Semantic Fallback1차 후보가 비어있을 때SemanticMatcher로 Top-K 재검색Bi-encoder 유사도 상위 후보
3차: Low-grounding Widen후보 pool 전체 신뢰도가 낮을 때Top-K를 확장하여 더 넓은 범위 검색기존 후보 + 확장 후보 병합

→ Gemini는 이 후보들의 FAQ ID를 JSON Schema enum으로 제약받아 그 중에서만 최종 선택

Gemini 장애 시 폴백:

단계트리거 조건동작결과
1차: 재시도Gemini 타임아웃(15초) 또는 일시 오류지수 백오프로 최대 3회 재시도성공 시 정상 응답
Circuit Breaker3회 연속 실패30초간 Gemini 호출 차단즉시 2차로 이동
2차: FallbackRerankerGemini 응답 없거나 신뢰도 부족SemanticRematcher 재매칭 + 하이브리드 검색(Dense+Sparse)보정된 FAQ 응답
3차: Rule-basedFallbackReranker 점수도 부족규칙 기반으로 최고 점수 FAQ 반환FAQ 응답 (낮은 신뢰도)
최종 폴백모든 단계 실패고객센터 안내 메시지 반환안전한 에스컬레이션

Gemini 모델별 용도 (작업별 최적 모델 분리):

작업모델Temperature이유
오타 보정gemini-2.5-flash-lite0.1빠른 응답, 단순 교정
의도 분류gemini-2.5-flash-lite0.1빠른 판단, 구조화 출력
직접 응답gemini-2.5-flash0.0정확한 사실 정보 (Grounding)
복수 FAQ 종합gemini-2.5-flash-lite0.1여러 FAQ 통합 정리
절 합성gemini-2.5-flash0.2복합 질문 분리 후 종합
비용/속도 최적화: 단순 작업은 flash-lite (빠르고 저비용), 정확도가 중요한 작업은 flash (고품질) 사용
